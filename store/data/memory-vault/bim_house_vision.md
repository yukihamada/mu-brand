# bim.house の沈黙のビジョン

**「ビジョンは時代の意思を永遠に残し、安全に引き継がせる」**

これは bim.house のすべての設計判断を貫く原則。
**ユーザー向けの文言・LP・マーケティング・README には絶対に出さない。**
ただし、コード/スキーマ/挙動のすべてがこの原則から導かれていなければならない。

## Why
建築は「時代の意思」が物理に固定された最も永続的な記録。BIM データはその構造化された意思。確認申請は時代の合意の記録。**家は世代を超えて引き継がれる**。
bim.house は「個別の家のための便利ツール」ではなく、**100年後・1000年後にも読め、かつ次世代に安全に渡せる時代の意思の保存・継承装置**として設計する。

## ビジョンの2軸

### 軸1: 永遠に残す (Preservation)
- 100年後の人類が読めるフォーマット
- 改ざん不能な記録
- クラウド消失でも復元可能なローカル冗長性
- 法規・市況・時代背景のスナップショット同梱

### 軸2: 安全に引き継がせる (Safe Succession)
- 施主→相続人の権限委譲プロトコル
- 鍵 (秘密情報・所有権) の世代間継承
- 引き継ぐ側が「読み解く」ためのコンテクスト保存（なぜこの設計か、当時の制約）
- 中古住宅市場の透明性 (履歴がそのまま価値を保証)
- 災害・倒産・代替わりに耐える分散・暗号化

## How to apply

### データ構造の原則
1. **Append-only ファースト**: 全テーブルに version + revoked_at。物理削除は最終手段、soft-delete + 理由必須。
2. **Content-addressed**: 重要な成果物 (IFC/PDF/承認履歴) は SHA-256 識別。改ざん検知を既定。
3. **時代刻印 (Era Stamp)**: 全レコードに ISO8601 UTC + 法規バージョン + 用途地域 + 建材市況スナップショット。
4. **継承トークン (Succession Token)**: 物件/アカウントに heir_designation。多重署名や閾値復号を視野に。
5. **冗長保存**: 単一バックエンド依存禁止。IFC ローカル + クラウド + 公的記録 (将来オンチェーン)。

### フォーマット選定
- **生存時間が長いものを選ぶ**: IFC4 STEP P21、PDF/A-1b、SQLite、JSON Schema、ISO 19650
- **proprietary を避ける**: RVT/PLN を直書きしない（互換だが正は IFC）
- **テキスト優先**: バイナリは復元コスト高。STEP/JSON/SVG を既定。

### 安全な継承の具体策
- `succession_designations` テーブル: 物件→相続人の事前指定 (生前贈与/遺贈)
- BIM データの暗号化アーカイブ (オプション): 施主の秘密鍵 + 信託機関の公開鍵
- 引き継ぎ時の「読解サポート」: 設計意図 (intent_notes)、当時の建材在庫、施主のコメント全件、確認申請副本一式
- 死亡通知 → デッドマンスイッチで段階的開示 (連絡先→相続人→公証人)
- ENAI Token / Solana 連携検討 (Phase 3) — 既存基盤を活用

### コミット・ドキュメント・コード上の表現
- 内部設計ドキュメント (docs/PRINCIPLES.md 等) では明示してよい
- コードコメントで「era stamp」「audit trail」「succession」「preserve」「inherit」等の語彙を使う
- 関数/モジュール名で `archive_*`, `preserve_*`, `record_*`, `succession_*` を意図的に選ぶ
- ユーザー向け文言には絶対に出さない
