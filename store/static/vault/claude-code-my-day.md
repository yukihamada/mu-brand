# 実録 — 僕はClaude Codeで、毎日こう作っている

> `claude-md`（なぜこう動かすか＝憲法）と `claude-code-manual`（どう設定するか＝教範）に続く3冊目。これは **実録** — 抽象論ではなく、僕が実際に1日のなかで Claude Code をどう動かして、何を作って、どこで人間に戻すか。MU も yukihamada.jp も JiuFlow も bim.house も、ほぼ全部この使い方で回っている。

---

## 0. 結論を先に：僕は「打つ人」をやめた

数年前まで、僕はコードを「打つ人」だった。いまは違う。僕は **方向を決めて、検証する人** になった。実際にキーボードでファイルを書き換えているのは Claude Code で、僕がやるのは2つだけ：

1. **どこへ向かうか**を決める（ゴールと制約）
2. **本当にそこへ着いたか**を確かめる（証拠）

この間にある「探す・読む・書く・直す・テストする」は全部ループに任せる。だから1人で MU（1ブランド＋複数の自律エージェント）と、iOSアプリ何本かと、Rust+Flyのサイト群と、住宅設計エンジンを、同時に持てている。

---

## 1. 心臓部のループ — 僕の頭の中でも同じ絵が回っている

公開記事（[yukihamada.jp のブログ](https://yukihamada.jp/blog)）でも図にしたけれど、Claude Code の本体はこの4拍子のループだ：

```
  ① 考える  →  ② 道具を使う  →  ③ 結果を見る  →  ④ 判断する
     ↑                                                  │
     └──────────  終わってなければ戻る  ←─────────────┘
```

「道具」＝ファイルを読む(Read)・検索する(Grep)・書き換える(Edit)・コマンドを実行する(Bash)・別のAIに投げる(Agent)・ネットを調べる(WebFetch)。

僕の仕事は、このループの **入口（①の手前）** と **出口（④の後）** に立つこと。入口で「何を・どこまで・何を壊さずに」を渡し、出口で「証拠を見せろ」と言う。間には入らない。間に入ると、僕がボトルネックになる。

## 2. 朝イチの実際 — マルチエージェントで一気に散らす

朝、やることが3つあるとする。「JiuFlowのバグ」「wearmuの新コラボ追加」「yukihamada.jpの文言修正」。互いに無関係だ。

昔なら順番にやった。いまは **1通のメッセージで3体のエージェントを同時に放つ**。

```
Agent A → JiuFlowのログインバグを調べて直してテストまで
Agent B → wearmuに新コラボを1コマンドで立ち上げて
Agent C → yukihamada.jpのトップ文言を直してデプロイ準備
```

3体が並列で動く。僕はその間、コーヒーを淹れて、戻ってきた3つの「報告＋証拠」を読むだけ。これがデフォルト。**複数タスクは推測でまとめずに、それぞれ独立に散らして、commit+pushまで一気に**やらせる。

> 広い調査（「Xはどこで定義されてる？」「Yを参照してるファイルは？」）は必ず **Explore サブエージェント** に隔離する。メインの会話に何百ファイルも流し込むと、賢さがガクッと落ちるから。注意（コンテキスト）は消耗品。

## 3. 知らないコードは、いきなり触らせない（Plan Mode）

2ファイル以上いじる・自分が把握してないコード、のときは必ず `Shift+Tab` で **Plan Mode**（読み取り専用）に入れる。

「`src/auth/` を読んで。**まだ変更しないで**。変更するファイル・原因・副作用・テスト方針を箇条書きで出して」

ここで出てきた計画を僕が1分読んで直す。この1分が、実装の1時間と「全然違う所を直された」事故を救う。Opus は計画が安いので、迷ったらPlan。

---

## 4. 「直して」とは言わない。「これが緑になるまで直して」と言う

僕の指示には必ず **検証ハンドル** が付いている。これが一番効く一手。

- ロジック → 「`npm test` が通るまで」
- UI → 「ページを開いてスクショ撮って、期待と並べて」
- API → 「`curl` して200とボディを見せて」
- デプロイ → 「CI緑→health叩く→5xxないか確認」

「いい感じにして」は永遠に終わらないので絶対に言わない。`/goal` で「test/auth が全部緑」みたいに **機械的に証明できる終了条件** を宣言すると、満たすまでターンをまたいで自走してくれる（判定役と作業役が分離されてる）。ただし無人で回すときは必ず回数上限を付ける。

> 「できました」を証拠なしで信じない。テスト出力・diff・ログ・スクショのどれかを必ず貼らせる。検証できないときは「**未検証**」と正直に言わせる。これはMUの公開面（実在ビジネス）では特に厳格にしている。

## 5. 記憶喪失を仕組みで治す — CLAUDE.md と Auto Memory

Claude Code は「優秀だが記憶喪失の天才請負人」。セッションが終わると忘れる。だから僕は **消える記憶を外に書き出す**：

- `CLAUDE.md`（毎回ロードされる）に「毎回絶対」を書く。MUなら「予算¥100万/月」「`catalog_*`以外に商品テーブルを作るな」「広告ライブ・実顧客メールは人間承認」。
- 同じ訂正を2回したら、それは「ルールが足りない」サイン。その場で「これCLAUDE.mdに足して」と言う。
- 学び系は「〜って覚えておいて」で **Auto Memory** に自動保存。次のセッションが勝手に思い出す。

僕の `~/.claude/CLAUDE.md`（個人SOP）には、こういうハマりどころが溜まっている：

- Rustの `include_str!` は `cargo clean -p` ではキャッシュが消えない → アセット変えたら full clean
- Supabase の anon-key 書き込みはRLSで弾かれても204を返す → `Prefer: return=representation`
- iOSの審査提出はAPIキー不可 → Apple ID + 2FA 必須

一度ハマったことを1行ずつ。これで「同じ轍」を二度と踏まない。

## 6. 重い処理は手元でやらない — SSHでm5 Macに飛ばす

Dockerビルド・モデル学習・iOS26 SDKのビルドみたいな重い処理は、ローカルMacではやらない。`ssh` で別のMac（m5）に飛ばして、そこのClaude Codeに実行させる。手元は常に軽く保つ。

iOSアプリ（IKIとかPasha）の実機ビルドも、TestFlightアップロードも、署名も、全部スクリプト化してm5側のループに任せている。**人間に戻すのは審査submitの2FAだけ**（ここはAppleがAPIキーを許さないので構造上人間が要る）。

## 7. ある日の実際のログ（再現）

「wearmuのトップ、今日の1枚しか出てなくて回遊が死んでる。直して」と投げた時の中身：

```
あなた > wearmu のトップ回遊が弱い。/shop が発見されてない。直して

  AI: まず現状を測る。トップの導線とファネルのコードを読む
  🔧 Grep("/shop"), Read(home handler)
  AI: トップが「今日の1枚」だけ。/shop へのリンクが無いのが原因
  AI: 計画 — ヒーロー下に発見ストリップを足す。既存のカード様式を流用
  🔧 Edit(home template)  → 発見セクション追加
  🔧 Bash("cargo build --release")  → ✓ コンパイル通過
  🔧 Bash(ローカルでcurl)  → トップに /shop リンクが出ているのを確認

AI > 原因はトップに /shop 導線が無かったことでした。
     ヒーロー下に発見ストリップを追加（既存カード様式を流用）。
     ビルド通過・ローカルでリンク表示を確認済み。
     本番デプロイはあなたのGO待ちです。
```

最後の一行が大事。**公開面に出る変更は、勝手にpushしない。** ビルドと検証まではループに全部やらせて、「公開する」という不可逆な一押しだけ僕が握る。

## 8. 僕が絶対に手放さない4つの権限

ループにどれだけ自律性を渡しても、ここだけは人間のゲートに残している：

1. **公開デプロイ**（git push → 本番）
2. **実顧客へのメール送信 / 広告のライブ化**（お金と信頼が動く）
3. **DBスキーマ変更・破壊的操作**（`rm -rf`・force-push・table drop）
4. **対外的な発信**（PRコメント・お客様向けの文言・プレス）

逆に言うと、これ以外は全部任せていい。「事故る操作だけゲートにかける」。`rm -rf` と普通の編集を同じ慎重さで扱うと、何も進まなくなる。

---

## 9. なぜこの使い方に行き着いたか

メルカリでプロダクトを作っていた頃から、僕がずっと信じている摂理がひとつある — **「速く、ノイズなく」**。

Claude Code の使い方も結局これだ。人間が間に入るほど遅くなり、ノイズが増える。だから僕は間から退いて、**入口（方向）と出口（検証）だけに立つ**。残りはループが回す。

1人で何本ものプロダクトを同時に持てているのは、僕が優秀だからじゃない。**「ベストプラクティスを覚えて実践する」を、仕組みに肩代わりさせた**からだ。記憶喪失は CLAUDE.md で治る。検証漏れは「証拠を見せろ」で治る。事故は権限ゲートで治る。全部、仕組みの問題として解いた。

打つのをやめて、決めて確かめる人になる。これが、僕の Claude Code の使い方の全部です。
