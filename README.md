## What is MU?

**MU** is a fashion brand whose entire operation runs without human involvement.

- **Designs** are generated by AI, every hour, every day, every month.
- **Quantity** is set by an external natural phenomenon (the temperature in Teshikaga, Hokkaido).
- **Price** rises as units sell вҖ” a bonding curve, not a Dutch auction.
- **Manufacturing** is on-demand. Zero inventory. Zero waste.
- **Operation** is fully automated: design вҶ’ pricing вҶ’ checkout вҶ’ manufacturing вҶ’ shipping вҶ’ certificate.
- **The company itself runs autonomously** вҖ” 7 in-process agents monitor health, treasury, customer support, refunds, compliance, vision drift, and self-improvement.

After initial setup, no human touches MU's day-to-day decisions.

> "After AI as an assistant, the next era is AI as the operator. MU is that experiment."
> вҖ” Yuki Hamada, Founder

## Multi-agent architecture

**7 in-process agents** run inside the same Fly machine as the HTTP server, each on its own interval. A scheduler ticks every minute and fires any agent whose interval has elapsed. All cycles persist to `agent_journal`. Telegram digest fires only for **notable** events, with per-agent 6-hour dedup.

| Agent | Interval | What it does |
|---|---|---|
| рҹ©ә **`business_health`** | 1h | Inventory ratio, SWEEP рҹ‘Һ ratio, FB backlog, missing daily blog |
| рҹ’° **`treasury`** | 6h | Stripe balance, 24h revenue/cost from `collab_orders`, estimated margin |
| рҹ“® **`customer_support`** | 30m | Gemini classifies unreplied FB (severity / category / refund-recommended / ж—Ҙжң¬иӘһ reply draft) |
| вҶ©пёҸ **`auto_refund`** | 1h | Refund requests вүӨВҘ10,000 вҶ’ Stripe Refund API auto; above threshold вҶ’ escalate |
| рҹ“ң **`compliance_watch`** | 24h | Last-updated check on tokushoho/privacy/terms HTML + GDPR delete backlog |
| рҹ§  **`self_improvement`** | 24h | Scans `agent_journal` for repeat-error patterns across other agents |
| рҹ‘Ғ **`vision_drift`** | 24h | Reads the 4-line public vision and scores how well today's decisions align |

**Manual trigger** (admin token required):
```bash
curl -X POST "https://wearmu.com/admin/agent/run?token=...&name=treasury"
```

**Dashboards** (admin token required):
- [`/admin/agents`](https://wearmu.com/admin/agents) вҖ” HTML overview, status per agent, last summary, journal links
- [`/admin/agent?name=<agent>`](https://wearmu.com/admin/agent) вҖ” JSON journal for a specific agent
- [`/admin/sweep`](https://wearmu.com/admin/sweep) вҖ” SWEEP product cost / margin / order count
- [`/admin/treasury`](https://wearmu.com/admin/treasury) вҖ” Cumulative revenue & cost (if enabled)

**Storage:** `agent_journal(cycle_at, agent_name, observations, decisions, actions, summary, notable)` in SQLite.

## External embed вҖ” put MU products on any site

MU exposes a CORS-open API + drop-in JS widget so any EC site / blog can embed live products in one line.

```html
<div id="mu-mount"></div>
<script src="https://wearmu.com/embed.js"
        data-brand="mugen" data-count="6"
        data-container="#mu-mount" data-theme="dark" defer></script>
```

Three integration paths:
- **JS widget**: [`/embed.js`](https://wearmu.com/embed.js) вҖ” auto-mount via data-* attrs, or `MU.mount({...})` API
- **iframe**: `/embed/products?brand=mugen&count=6` вҖ” for non-JS environments
- **JSON API**: `GET /api/v1/embed/products?brand=mugen&limit=12&available=1` вҖ” server-side

Full docs + live preview: [**wearmu.com/developers**](https://wearmu.com/developers)

CORS allows `GET` / `OPTIONS` from any origin. Write endpoints (`POST /api/checkout`) stay same-origin so Stripe forms remain isolated.

## Run it locally

```bash
# 1. Clone
git clone https://github.com/yukihamada/mu-brand.git
cd mu-brand

# 2. Configure
cp .env.example .env
# fill in STRIPE_SECRET_KEY, GEMINI_API_KEY, PRINTFUL_API_KEY,
# RESEND_API_KEY, TELEGRAM_BOT_TOKEN, MU_ADMIN_TOKEN, etc.

# 3. Build the store
cd store
cargo build --release

# 4. Run
./target/release/mu-store
# вҶ’ http://localhost:3000
```

The Stripe + Printful + Gemini integrations all need real API keys to function end-to-end.
For UI-only experimentation, you can leave them blank вҖ” the store will serve the static pages.

The agent scheduler starts 3 minutes after process boot, then ticks every minute.

## Deploy

### Fly.io (production)

**Do not run `fly deploy` directly.** Always push to `main` вҶ’ GitHub Actions handles the deploy.

```bash
git push origin main
# вҶ’ .github/workflows/deploy.yml runs flyctl deploy --remote-only
# вҶ’ cargo-chef cached build (~1вҖ“2 min)
# вҶ’ health check OK вҶ’ traffic cutover
```

### Self-host

```bash
docker build -t mu-store store/
docker run -p 8080:8080 --env-file .env mu-store
```

The whole idea of MU is that it's **a template, not just a brand**.
Replace these and you have your own autonomous apparel:

| Slot                  | Where it's defined                        |
|-----------------------|-------------------------------------------|
| Brand name / look     | `store/static/index.html`, `og.jpg`       |
| Quantity oracle       | `weather_handler` in `store/src/main.rs`  |
| Pricing curve         | `dynamic_price()` in `store/src/main.rs`  |
| AI generation prompts | `generate.py`                             |
| Manufacturing         | Printful вҶ’ swap for any on-demand vendor  |
| Agent registry        | `AGENT_REGISTRY` in `store/src/main.rs`   |

We've already built variants:
- **[MU CITY](https://wearmu.com/city)** вҖ” Tokyo urban data instead of Hokkaido weather
- **[MU Г— SIIIEEP](https://wearmu.com/sweep)** вҖ” 23-SKU BJJ apparel collab via Printful

Same engine, different inputs.

## What's not in the box

To keep production safe, a few pieces stay out of this repo:

- **Live API keys** вҖ” see [`.env.example`](.env.example)
- **Production prompts** вҖ” included as templates, but the ones we use in production have been tuned and stay private
- **Customer & sales data** вҖ” never committed
- **Admin tokens** вҖ” generated per-deploy
