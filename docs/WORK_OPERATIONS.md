# WORK OPERATIONS — 応募が来たら何をするか（梱包ワーカー運用SOP）

wearmu.com/work の応募〜報酬振込までの全手順。**太字は人間ゲート**（自動化しない判断箇所）。
コードの実体は `store/src/work.rs`。広告・価格A/Bの経緯は auto-memory `mu_work_recruitment` / `adflow_zltv_self_improving_loop`。

## 全体フロー

```
応募(/work) ─→ Gemini判定+m5ダイアログ ─→ 【承認】 ─→ 自動メール(キューURL+住所返信依頼)
                                                      │
   ┌──────────────────────────────────────────────────┘
   ▼
ワーカーが住所返信 ─→ 【キット郵送】 ─→ 注文発生 ─→ キューに表示 ─→ claim(早い者勝ち)
                                                      │
   ┌──────────────────────────────────────────────────┘
   ▼
作業(検品→包む→カード→投函) ─→ ship(追跡番号+写真URL) ─→ 顧客に自動メール+telegram通知
                                                      │
   ┌──────────────────────────────────────────────────┘
   ▼
月初: /admin/work/payouts で集計 ─→ 【振込先確認→銀行振込】 ─→ /admin/work/mark_paid
```

## 1. 応募が来たら（数分以内・ほぼ自動）

- telegram「🧵 work応募」+ m5ダイアログが鳴る（watcher `tokyo.hamada.work-watcher`）。
- 通知には **パターンv・提示単価(¥200/¥300)・名前・地域・自己紹介・GeminiのSAFE/SKETCHY判定** が載る。
- **承認 or 却下を人間が押す**。判断基準:
  - SAFE + 自己紹介に体温がある → 承認
  - SKETCHY（捨てメール・金銭要求・bot文面）→ 却下
  - 迷ったら承認でOK（住所・商品はキット送付まで渡らないので被害半径は小さい）
- 承認すると自動で: status=active・専用トークン発行・キューURLをメール送付・**住所の返信依頼**と振込タイミング案内も同メールに入っている。

## 2. 住所が返ってきたら（24h以内目標・人間）

- **キット（資材一式）を郵送する**。中身:
  - Tシャツ手仕上げ用: 薄紙・MU封緘シール・一言カード(テンプレ)・クリックポスト用封筒
  - 音コイン用: ブランクNFCコイン・封筒・宛名シール
- 住所はキット送付のみに使い、**メモリ・コード・公開ファイルに書かない**（応募同意の建付けと同じ）。
- 資材の在庫が無い場合: 発注は人間ゲート（¥30,000超は BUDGET.md §3 のゲート対象）。

## 3. 仕事の発生（自動 + Phase 2は人間トリアージ）

### いま自動で流れるもの: 音コイン
- `fulfillment_route='manual'` の注文が paid になると `manual_pending` でキューに出る。
- ワーカーが claim → 全住所表示 → NFC書込 → 検品 → 発送 → 追跡番号+完成写真URL入力。
- 顧客への発送完了メールは自動。**運営は telegram の📮通知と📷写真を確認するだけ**（変な梱包なら本人に連絡）。
- 単価: 音コインは `fee_jpy()`（固定）。

### Phase 2（未実装・設計のみ）: Tシャツの手仕上げ中継
広告の主訴求「Tシャツを包んで送る」はまだ自動ルーティングされない。実装するまでの暫定運用:
- 注文が少ない立ち上げ期は、**手仕上げにしたい注文を運営が選び、Printfulの送付先を手動でワーカー住所に変更**（Printful管理画面）→ ワーカーに「届いたら包んでこの住所へ」をメールで個別依頼 → ship報告はキューでなくメール/telegramで受ける → 報酬は `work_assignments` に手動INSERTせず、月次集計時にメモで加算（件数が月10件を超えたら Phase 2 を実装する）。
- Phase 2 実装条件: ワーカー2人以上 & 手仕上げ依頼が月10件超。実装内容: ワーカー住所のDB管理(同意文言の更新込み)・Printful recipient差し替えフック・Tシャツジョブのキュー表示(単価=worker.rate_jpy)。

## 4. 単価（価格A/Bの約束を守る）

- 応募時に見た単価が `work_workers.rate_jpy` に保存される（v奇数=¥200 / v偶数=¥300）。
- **¥300を見て応募した人のTシャツ系ジョブ単価は¥300前後にする**（見せた金額は約束）。音コインは従来どおり固定。
- A/B判定: `/admin/work/payouts` の rate_jpy 別に応募数・稼働数を比較。LP側CVRは `work_view`/`work_apply`（price プロパティ・パターン偶奇でも導出可）。

## 5. 月次振込（毎月1〜5日・人間）

1. `GET /admin/work/payouts?token=ADMIN_TOKEN&month=YYYY-MM`（前月分）→ ワーカー別の未払い件数・合計JPY。
2. 初回のワーカーには**メールで振込先口座を聞く**（このタイミングまで口座情報は預からない）。
3. **銀行振込を実行**（振込手数料は当社負担とLPで約束済み）。
4. `GET /admin/work/mark_paid?token=…&worker_id=<id>&month=YYYY-MM` で台帳に振込済みマーク（未払い分のみ更新・冪等＝二重払い防止）。
5. telegramに💴確認が流れる。

## 6. 日次チェック（kuberu に登録済み）

- 広告: `adflow_metrics google 5408218744` + 新RSA審査ステータス（REJECTED即対応）
- 応募: `/admin/work/pending?token=` 件数
- watcher死活: `launchctl list | grep work-watcher`
- 価格A/B: 応募が10件溜まったら ¥200/¥300 のCVR比較を一度見る

## 禁止・注意

- 承認の自動化はしない（human-gate維持）。welcome-set自動発送も未実装のまま。
- ワーカー住所・顧客住所を memory / コード / 公開ファイルに書かない。
- 報酬の約束（提示単価・手数料当社負担・翌月振込）はLPに明文化されている＝変更は本人判断+LP更新とセット。
